# Hypagraph v0.5 dogfood record

- Status: release candidate evidence
- Milestone: M4
- Release: v0.5.0
- Product path: Pi extension tools, commands, persistence, graph projection, and hosted platform matrix

## Purpose

This record maps the v0.5 release requirements to executable product-path evidence. The evidence uses the same Pi extension entry points, reducer, event store, check runner, restore path, graph projection, and status projection that the installed package uses.

The record does not treat repair as the default loop purpose. The scenarios cover optimization, bounded processing, repeated evaluation, independent regions, and check-and-repair.

## Required workflows

### Refinement and optimization

Evidence:

- `tests/loop-progress.test.ts`
- `tests/pi-loop-progress.test.ts`
- `tests/loop-product-surface.test.ts`

The workflow records an initial metric, accepts later improvements, retains the best metric and iteration, applies patience, and exits independently from the progress decision when the typed success condition becomes true.

### Bounded batch processing

Evidence:

- `tests/loop-slice-one.test.ts`
- `tests/loop-slice-two.test.ts`
- `tests/loop-product-surface.test.ts`

The region uses typed facts at the evaluation boundary, follows only declared feedback, clears current iteration facts and routes, retains history, and exits when its typed completion condition becomes true.

### Check-and-repair as one loop pattern

Evidence:

- `tests/loop-check-repair.test.ts`
- `tests/loop-max-iterations-check.test.ts`
- `tests/pi-loop-limit.test.ts`

A failed command check can publish a complete observation. The loop can continue from that observation without making repair a special loop type. A later passing observation can complete the region. The hard limit remains deterministic.

### Two disconnected loop regions

Evidence:

- `tests/loop-outcome-policy.test.ts`
- `tests/pi-loop-outcome-policy.test.ts`
- `tests/graph-projection.test.ts`

The scenarios prove that both independent entries can become ready, work can be interleaved under the one-active-attempt rule, one region can complete while another continues, and `record-and-continue` does not terminate unrelated work. Facts, routes, attempts, progress, resets, and graph components remain isolated.

## Product-surface evidence

The v0.5 surface exposes:

- current iteration and hard limit;
- evaluation boundary;
- selected feedback;
- current and best metrics;
- best iteration;
- no-progress count and remaining patience;
- failure policy;
- independent component identity;
- local region outcome and wider workflow effect;
- explicit exit reason and warnings.

The graph pane remains read-only. Navigation cannot select or change a loop decision.

## Platform gate

The authoritative `CI` workflow runs `npm run check` on:

- Ubuntu with Node.js 22 and 24;
- macOS with Node.js 22 and 24;
- Windows with Node.js 22 and 24.

The release branch must pass all six jobs after the final version and documentation commit. The v0.5.0 tag must point to the tested main commit after merge.
